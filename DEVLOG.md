# Dev Log

## 2026-07-15

Added three new API backends to the gateway — OpenRouter, Venice, and a generic self-hosted backend for local OpenAI-compatible servers (Ollama, llama.cpp, vLLM) — bringing it to provider parity with the sibling Python llm-backends package. Since all three speak the OpenAI chat-completions protocol, the existing OpenAI adapter's logic (complete/stream over a configurable base URL plus bearer key, SSE parsing, request-body build, finish-reason mapping) was extracted into a shared `OpenAiCompatibleApiAdapter` that the OpenAI adapter and the three newcomers now extend. OpenAI's own behavior is deliberately untouched: same default base URL, model list, and displayName, so even its error-string prefix stays byte-identical. The change also threaded the new backends through the factory, the `ApiBackend` union, DTO/validation/health backend lists, and env config, with `isApiBackend` now single-sourced from the factory.

**Decisions & notes:** Two design choices worth remembering. Venice needs `venice_parameters.include_venice_system_prompt=false` injected so the caller's prompts fully govern; rather than pollute the shared base, this goes through an overridable `extraBodyParams()` hook, keeping the base provider-neutral. The hosted backend's `isConfigured()` checks for a base URL rather than an API key, so keyless local servers work, and the factory guards it on base URL rather than key for the same reason. Env var names intentionally match the Python package so a single `.env` serves both stacks. Test count went 51 → 60. Surfacing these backends in the web UI/SDK (their own separate backend literals) is a deliberate follow-up, not compiler-forced.
